# Feature Specification: 다크모드

**Feature Branch**: `003-dark-mode`

**Created**: 2026-07-21

**Status**: Draft

**Input**: User description: "다크모드 기능을 구현해주세요. 사이드 바에 토글 버튼을 만들고, 이걸 클릭하면 다크모드로 변경되어야 합니다."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 사이드바 토글로 다크모드 전환 (Priority: P1)

사용자가 사이드바에 있는 테마 토글 버튼을 클릭하면 앱 전체 화면이 즉시 다크모드로
바뀐다. 다시 클릭하면 라이트모드로 돌아온다. 어두운 환경에서 작업하는 사용자가
눈의 피로를 줄이고 편안하게 글을 읽고 쓸 수 있다.

**Why this priority**: 이 기능의 핵심 가치 전체가 여기에 있다. 토글 한 번으로
다크모드가 켜지고 꺼지는 것만 동작해도 사용자는 요청한 목적을 완전히 달성한다.

**Independent Test**: 사이드바의 토글 버튼을 클릭했을 때 화면 전체(사이드바, 목록,
편집 화면)가 어두운 테마로 바뀌고, 다시 클릭하면 밝은 테마로 돌아오는지 확인하는
것만으로 독립적으로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 라이트모드 상태의 앱, **When** 사용자가 사이드바의 테마 토글 버튼을
   클릭하면, **Then** 페이지 이동이나 새로고침 없이 앱 전체 화면이 즉시 다크모드로
   전환된다.
2. **Given** 다크모드 상태의 앱, **When** 사용자가 토글 버튼을 다시 클릭하면,
   **Then** 앱 전체 화면이 즉시 라이트모드로 돌아온다.
3. **Given** 어느 테마 상태든, **When** 사용자가 토글 버튼을 보면, **Then** 버튼이
   현재 테마 상태(또는 전환될 대상)를 시각적으로 알 수 있게 표시한다.
4. **Given** 다크모드 상태, **When** 사용자가 게시글 목록·편집 등 다른 화면으로
   이동하면, **Then** 이동한 화면도 다크모드로 표시된다.

---

### User Story 2 - 선택한 테마 유지 (Priority: P2)

사용자가 선택한 테마는 저장되어, 페이지를 새로고침하거나 나중에 다시 방문해도
마지막으로 선택한 테마가 그대로 적용된다.

**Why this priority**: 방문할 때마다 다크모드를 다시 켜야 한다면 기능의 실질적
가치가 크게 떨어진다. 다만 전환 자체(US1)가 먼저 동작해야 의미가 있으므로 P2다.

**Independent Test**: 다크모드로 전환한 뒤 페이지를 새로고침했을 때 다크모드가
유지되는지 확인하는 것만으로 독립적으로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 사용자가 다크모드를 선택한 상태, **When** 페이지를 새로고침하면,
   **Then** 다크모드가 그대로 유지된다.
2. **Given** 사용자가 다크모드를 선택한 상태, **When** 브라우저를 닫았다가 나중에
   다시 방문하면, **Then** 다크모드로 표시된다.
3. **Given** 저장된 테마가 있는 상태, **When** 페이지가 로드되면, **Then** 반대
   테마가 잠깐 보였다가 바뀌는 깜빡임 없이 처음부터 저장된 테마로 표시된다.

---

### User Story 3 - 첫 방문 시 시스템 설정 따르기 (Priority: P3)

아직 테마를 한 번도 선택하지 않은 사용자에게는 기기(운영체제)의 다크모드 설정을
따라 초기 테마를 보여준다. 시스템을 다크모드로 쓰는 사용자는 별도 조작 없이
처음부터 어두운 화면을 만난다.

**Why this priority**: 있으면 첫인상이 좋아지는 편의 기능이지만, 토글(US1)과
유지(US2)가 없으면 의미가 없고, 없어도 사용자가 토글로 직접 선택할 수 있다.

**Independent Test**: 저장된 테마 선택이 없는 상태에서 시스템 다크모드 설정을 켜고
접속했을 때 다크모드로 표시되는지 확인하는 것만으로 독립적으로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 테마를 선택한 적 없는 사용자와 다크모드로 설정된 기기, **When** 앱에
   접속하면, **Then** 다크모드로 표시된다.
2. **Given** 테마를 선택한 적 없는 사용자와 라이트모드로 설정된 기기, **When** 앱에
   접속하면, **Then** 라이트모드로 표시된다.
3. **Given** 사용자가 토글로 테마를 직접 선택한 상태, **When** 기기의 시스템 테마
   설정이 바뀌어도, **Then** 사용자가 직접 선택한 테마가 우선 적용된다.

---

### Edge Cases

- 테마 저장소를 사용할 수 없는 환경(예: 저장 차단)에서도 토글에 의한 세션 내
  전환은 정상 동작해야 하며, 이 경우 재방문 시 초기 테마로 돌아가는 것은 허용된다.
- 사용자가 테마를 선택한 적 없는 상태에서 사용 중 시스템 테마가 바뀌면 앱 테마도
  그에 따라 바뀐다. 직접 선택한 뒤에는 시스템 변화에 영향받지 않는다.
- 다크모드에서도 스켈레톤 로딩, 비어 있는 목록, 에러 메시지 등 모든 상태 화면이
  판독 가능해야 한다.
- 커버 이미지 등 사진 콘텐츠는 테마와 무관하게 원본 그대로 표시되며, 그 주변
  UI(테두리·배경·스켈레톤)만 테마를 따른다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 사이드바에 테마 전환 토글 버튼이 항상 표시되어야 한다.
- **FR-002**: 토글 버튼 클릭 시 페이지 이동·새로고침 없이 앱의 모든 화면 요소
  (사이드바, 게시글 목록, 편집 화면, 로그인 화면 포함)가 즉시 다크모드와
  라이트모드 사이를 전환해야 한다.
- **FR-003**: 토글 버튼은 현재 테마 상태를 시각적으로 구분할 수 있게 표시해야 한다.
- **FR-004**: 사용자가 선택한 테마는 저장되어 새로고침·재방문 시 그대로 적용되어야
  한다.
- **FR-005**: 저장된 선택이 없는 첫 방문에서는 기기의 시스템 테마 설정을 초기
  테마로 사용해야 한다.
- **FR-006**: 사용자가 직접 선택한 테마는 시스템 테마 설정보다 우선해야 한다.
- **FR-007**: 페이지 로드 시 저장된(또는 시스템) 테마가 처음부터 적용되어, 반대
  테마가 잠깐 보이는 깜빡임이 없어야 한다.
- **FR-008**: 다크모드에서 모든 텍스트와 UI 요소가 충분한 명암 대비로 판독
  가능해야 한다.

### Key Entities

- **테마 설정(Theme Preference)**: 사용자가 선택한 표시 테마. 값은 "라이트" 또는
  "다크"이며, 선택한 적이 없으면 "미설정(시스템 따름)" 상태다. 브라우저(기기)
  단위로 저장되며 계정에 귀속되지 않는다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 사이드바가 보이는 어떤 화면에서든 클릭 1번으로 테마 전환을
  완료할 수 있다.
- **SC-002**: 토글 클릭 후 1초 이내에, 페이지 이동 없이 화면 전체에 새 테마가
  반영된다.
- **SC-003**: 테마를 선택한 사용자가 새로고침하거나 재방문했을 때 100% 같은 테마를
  다시 보게 된다.
- **SC-004**: 다크모드에서 앱의 모든 화면(목록, 편집, 로그인, 로딩·빈·에러 상태)을
  둘러봤을 때 판독 불가능한 텍스트나 구분 불가능한 UI 요소가 0건이다.

## Assumptions

- 지원 테마는 라이트/다크 2종이다. 별도의 "시스템 따름" 선택지를 토글 UI에
  노출하는 것은 범위 외이며, 시스템 설정은 사용자가 선택하기 전의 초기값으로만
  쓰인다.
- 테마 설정은 브라우저(기기)별로 저장한다. 로그인 계정 간 테마 동기화는 범위
  외다.
- 토글 버튼의 위치는 사이드바 내부이며, 사이드바가 표시되지 않는 화면에 별도의
  테마 전환 수단을 추가하는 것은 범위 외다.
- 다크모드 색상 값은 프로젝트 디자인 시스템 문서(`DESIGN.md`)에 함께 정의되어
  문서와 구현이 어긋나지 않게 유지한다(저장소 규칙에 따른 의존성).
