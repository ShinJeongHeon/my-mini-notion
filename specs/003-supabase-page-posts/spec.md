# Feature Specification: 계정 기반 페이지 게시글 저장

**Feature Branch**: `worktree-wt1` (spec: `003-supabase-page-posts`)

**Created**: 2026-07-21

**Status**: Draft

**Input**: User description: "페이지 게시글 기능 구현할 겁니다. 이미 기존에 로컬스토리지로 구현되어 있었는데, 제가 Supabase에 만든 page 테이블에 저장되도록 변경해주세요. 제가 만든 DB 테이블 구조는 변경하면 안 됩니다. 로그인된 유저만 등록 가능하고, 자신의 글만 조회 가능하고, 자신의 글만 삭제할 수 있어야 합니다."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 로그인 사용자의 게시글 등록과 계정 기반 보관 (Priority: P1)

로그인한 사용자가 새 게시글(페이지)을 만들면 그 글은 브라우저가 아니라 사용자 계정에 귀속되어 중앙 저장소에 보관된다. 브라우저를 바꾸거나 다른 기기에서 같은 계정으로 로그인해도 자신의 글 목록과 내용이 그대로 보인다.

**Why this priority**: 이 기능의 핵심 가치 — "브라우저에 갇힌 데이터"를 "계정을 따라다니는 데이터"로 바꾸는 것. 이것만 되어도 기능은 성립한다.

**Independent Test**: 로그인 → 새 글 등록 → 새로고침 및 다른 브라우저(또는 시크릿 창)에서 같은 계정으로 로그인 → 등록한 글이 목록과 상세에 그대로 보이면 통과.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자, **When** 새 게시글을 등록하면, **Then** 글이 목록 최상단에 나타나고 계정 저장소에 저장된다.
2. **Given** 글을 등록한 사용자, **When** 페이지를 새로고침하거나 다른 브라우저에서 같은 계정으로 로그인하면, **Then** 등록했던 글의 제목·본문·생성일이 동일하게 보인다.
3. **Given** 로그인하지 않은 방문자, **When** 게시글 등록을 시도하면, **Then** 등록이 차단되고 로그인이 필요함을 안내받는다.

---

### User Story 2 - 자신의 글만 조회 (소유자 격리) (Priority: P1)

사용자는 자신이 작성한 글만 볼 수 있다. 다른 사용자의 글은 목록에 나타나지 않고, 주소(URL)를 직접 입력해 접근해도 볼 수 없다. 로그인하지 않은 방문자에게는 어떤 글도 보이지 않는다.

**Why this priority**: 개인 메모 도구에서 소유자 격리는 보안·프라이버시의 최소 요건이며, 등록(US1)과 동시에 갖춰져야 안전하게 배포할 수 있다.

**Independent Test**: 계정 A로 글 등록 → 계정 B로 로그인 → B의 목록에 A의 글이 없고, A의 글 상세 URL로 직접 접근해도 내용이 보이지 않으면 통과.

**Acceptance Scenarios**:

1. **Given** 계정 A가 작성한 글이 있을 때, **When** 계정 B가 목록을 열면, **Then** A의 글은 목록에 나타나지 않는다.
2. **Given** 계정 A의 글 상세 주소를 알고 있는 계정 B, **When** 그 주소로 직접 접근하면, **Then** 글 내용이 노출되지 않고 "찾을 수 없음" 상태를 안내받는다.
3. **Given** 로그인하지 않은 방문자, **When** 글 목록 화면에 접근하면, **Then** 어떤 글도 보이지 않고 로그인이 필요함을 안내받는다.
4. **Given** 화면을 우회해 데이터에 직접 접근하려는 시도, **When** 다른 사용자의 글을 요청하면, **Then** 저장소 계층에서 차단되어 데이터가 반환되지 않는다.

---

### User Story 3 - 자신의 글 삭제 (Priority: P2)

사용자는 자신이 작성한 글을 삭제할 수 있고, 삭제된 글은 어디에서도 다시 보이지 않는다. 다른 사용자의 글은 어떤 방법으로도 삭제할 수 없다.

**Why this priority**: 데이터 관리의 기본 권한. 등록·조회가 갖춰진 뒤에 의미가 있다.

**Independent Test**: 자신의 글 삭제 → 목록·상세에서 사라지고 새로고침 후에도 돌아오지 않으면 통과. 타인 글 삭제 시도가 저장소 계층에서 거부되면 통과.

**Acceptance Scenarios**:

1. **Given** 자신이 작성한 글, **When** 삭제하면, **Then** 목록에서 즉시 사라지고 새로고침·재로그인 후에도 나타나지 않는다.
2. **Given** 다른 사용자의 글, **When** 화면을 우회해 삭제를 시도하면, **Then** 저장소 계층에서 거부되고 글은 유지된다.

---

### User Story 4 - 자신의 글 편집 유지 (Priority: P2)

사용자는 기존처럼 글 상세 화면에서 제목과 본문을 수정할 수 있고, 수정 내용은 계정 저장소에 반영되어 어디서 접속해도 최신 내용이 보인다. 편집 역시 자신의 글에만 가능하다.

**Why this priority**: 기존 앱이 제공하던 편집 경험의 연속성. 저장 위치가 바뀌어도 사용자가 체감하는 기능은 그대로여야 한다.

**Independent Test**: 자신의 글 제목·본문 수정 → 새로고침 또는 다른 브라우저에서 재로그인 → 수정된 내용이 보이면 통과.

**Acceptance Scenarios**:

1. **Given** 자신이 작성한 글, **When** 제목이나 본문을 수정하면, **Then** 수정 내용이 저장되어 새로고침·재로그인 후에도 유지된다.
2. **Given** 다른 사용자의 글, **When** 화면을 우회해 수정을 시도하면, **Then** 저장소 계층에서 거부되고 원본이 유지된다.

---

### Edge Cases

- 로그아웃 상태에서 글 등록·조회·삭제를 시도하면? → 모두 차단되고 로그인 안내가 표시된다.
- 저장(등록·수정·삭제) 중 네트워크가 끊기면? → 사용자에게 실패가 안내되고, 성공한 것처럼 보이는 상태로 남지 않는다.
- 이전 방식(브라우저 보관)으로 저장돼 있던 기존 글·샘플 글은? → 계정 저장소로 이관하지 않으며 더 이상 목록에 나타나지 않는다 (Assumptions 참조).
- 삭제된 글의 상세 주소로 다시 접근하면? → "찾을 수 없음" 상태를 안내받는다.
- 제목 없이(빈 제목) 글을 등록하면? → 기존과 동일하게 등록은 허용되며, 목록에서는 "제목 없음" 등 대체 문구로 표시된다.
- 기존 UI의 즐겨찾기(별표) 기능은? → 기능은 유지하되 즐겨찾기 표시는 사용 중인 기기(브라우저)에만 보관된다. 다른 기기에서는 즐겨찾기 표시가 공유되지 않는다 (FR-010).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 로그인한 사용자만 게시글을 등록할 수 있어야 한다. 로그인하지 않은 방문자의 등록 시도는 차단되고 로그인 안내가 표시되어야 한다.
- **FR-002**: 게시글은 작성자 계정에 귀속되어 중앙 저장소에 보관되어야 하며, 브라우저·기기가 바뀌어도 같은 계정으로 로그인하면 동일하게 조회되어야 한다.
- **FR-003**: 사용자는 자신이 작성한 글만 조회할 수 있어야 한다. 목록·상세 화면 모두에 적용되며, 로그인하지 않은 방문자에게는 어떤 글도 보이지 않아야 한다.
- **FR-004**: 사용자는 자신이 작성한 글만 삭제할 수 있어야 하며, 삭제는 계정 저장소에 반영되어 영구적이어야 한다.
- **FR-005**: 사용자는 자신이 작성한 글의 제목·본문을 수정할 수 있어야 하며, 수정 내용은 계정 저장소에 반영되어야 한다 (기존 편집 경험 유지).
- **FR-006**: 조회·삭제·수정의 소유자 제한은 화면 단이 아니라 저장소(데이터 접근) 계층에서 강제되어야 한다. 화면을 우회한 직접 접근으로도 타인의 글을 읽거나 변경할 수 없어야 한다.
- **FR-007**: 이미 마련되어 있는 게시글 저장 구조(항목: 식별자, 생성일시, 제목, 본문, 소유자)를 변경 없이 그대로 사용해야 한다. 항목 추가·삭제·변경은 허용되지 않는다.
- **FR-008**: 저장·삭제·수정 작업이 실패하면 사용자에게 실패가 안내되어야 하며, 화면 상태가 실제 저장 상태와 어긋난 채 남아 있으면 안 된다.
- **FR-009**: 신규 사용자의 첫 목록은 비어 있어야 한다. 이전 방식에서 제공되던 샘플 글 자동 생성은 하지 않는다.
- **FR-010**: 즐겨찾기(별표) 기능은 유지하되, 즐겨찾기 표시는 사용 중인 기기(브라우저)에만 보관한다. 계정 저장 구조에는 즐겨찾기 항목을 추가하지 않으며(FR-007), 즐겨찾기 표시가 기기 간 공유되지 않는 것은 허용된 동작이다.

### Key Entities

- **게시글(페이지)**: 사용자가 작성하는 글. 식별자, 제목(비어 있을 수 있음), 본문(비어 있을 수 있음), 생성일시, 소유자(작성자 계정)로 구성된다. 저장 구조는 이미 확정되어 있으며 변경 불가.
- **사용자 계정**: 기존 Google 로그인으로 식별되는 계정. 게시글의 소유자이며, 한 계정이 여러 글을 가질 수 있다(1:N).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 로그인한 사용자가 등록한 글이 새로고침 후와 다른 브라우저에서의 재로그인 후 모두 100% 동일하게 조회된다.
- **SC-002**: 다른 사용자 계정으로는 목록·상세·직접 접근 어떤 경로로도 타인의 글을 조회·수정·삭제할 수 없다 (우회 접근 성공 0건).
- **SC-003**: 로그인하지 않은 방문자는 글 등록·조회·삭제를 일절 수행할 수 없다.
- **SC-004**: 등록·수정·삭제 결과가 일반적인 네트워크 환경에서 2초 이내에 화면에 반영된다.
- **SC-005**: 기존 사용자 흐름(목록 보기 → 글 열기 → 편집 → 돌아오기)이 저장 방식 변경 전과 동일하게 완료 가능하다 (기능 회귀 0건).

## Assumptions

- **편집 기능 유지**: 사용자 요청에는 등록·조회·삭제만 명시되었으나, 기존 앱의 핵심 경험인 제목·본문 편집은 유지하며 소유자 본인에게만 허용하는 것으로 간주한다.
- **기존 데이터 이관 없음**: 이전 방식(브라우저 보관)의 글은 데모용 샘플 데이터이므로 계정 저장소로 이관하지 않는다. 계정 저장소는 빈 상태에서 시작한다.
- **샘플 글 자동 생성 중단**: 첫 방문 시 샘플 글을 만들어주던 동작은 계정 기반 저장에서는 수행하지 않는다.
- **로그인 수단 재사용**: 기존에 구축된 Google 로그인을 그대로 사용하며, 이번 작업에서 인증 방식은 변경하지 않는다.
- **프로필(닉네임·아바타) 범위 밖**: 닉네임·아바타 저장은 기존 동작을 유지하며 이번 변경 범위에 포함되지 않는다.
- **온라인 사용 전제**: 계정 저장소 접근에는 네트워크 연결이 필요하며, 오프라인 사용은 지원 범위 밖이다.
- **즐겨찾기는 기기별 보관** (사용자 확정, 2026-07-21): 즐겨찾기 표시는 계정이 아닌 기기(브라우저)에 보관한다. 기기 간 공유되지 않으며, 계정 저장 구조에는 어떤 항목도 추가하지 않는다.
