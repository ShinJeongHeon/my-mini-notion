  시 새 랜덤 이미지를 가져온다(게시글별로 저장·고정하지 않음). 같은 게시글을 다시 열면
  다른 사진이 보일 수 있다.
- Q: 이미지가 로드 실패(에러)는 아니지만 끝내 오지 않거나 매우 느릴 때 스켈레톤을 어떻게

  타임아웃은 두지 않으며, 로딩이 지연되는 동안에는 스켈레톤을 그대로 유지하되 제목·본문
  편집은 즉시 가능하다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 편집 화면 상단에서 커버 이미지 보기 (Priority: P1)

게시글 편집 화면을 연 사용자가 제목 입력창 바로 위에서 랜덤 고양이 커버 이미지를
확인할 수 있다. 커버는 화면 상단을 넓게 채워 문서에 시각적 인상과 분위기를 더한다.

**Why this priority**: 이 기능의 핵심 가치 전체가 여기에 있다. 이 하나만 동작해도
사용자는 원하는 목적(편집 화면 상단의 커버 이미지)을 완전히 달성한다.

**Independent Test**: 게시글 상세(편집) 화면을 열었을 때 제목 입력창 위쪽에 고양이
이미지가 표시되는지 확인하는 것만으로 독립적으로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 게시글 상세 화면, **When** 사용자가 화면을 열면, **Then** 제목 입력창
   바로 위에 커버 이미지 영역이 나타나고 로드가 끝나면 고양이 사진이 표시된다.
2. **Given** 커버 이미지가 표시된 상태, **When** 사용자가 화면을 아래로 스크롤하면,
   **Then** 커버는 제목·본문과 함께 문서 상단 콘텐츠로서 자연스럽게 스크롤된다.
3. **Given** 커버 이미지가 표시된 상태, **When** 사용자가 제목이나 본문을 입력하면,
   **Then** 커버 표시는 편집 동작을 방해하지 않고 그대로 유지된다.

---

### User Story 2 - 로딩 중 스켈레톤으로 자리 유지하기 (Priority: P2)

커버 이미지를 아직 내려받는 중일 때, 사용자는 회전하는 로딩 스피너 대신 커버 자리를
그대로 차지하는 스켈레톤(placeholder) UI를 본다. 이미지가 도착하면 스켈레톤이 실제
사진으로 매끄럽게 대체된다.

**Why this priority**: 커버는 외부 API에 의존해 즉시 표시되지 않을 수 있다. 스켈레톤이
없으면 이미지가 뒤늦게 나타나며 레이아웃이 밀려(layout shift) 제목·본문 위치가 튀는
불쾌한 경험이 발생한다. 핵심 표시(P1)를 안정적으로 만드는 UX 안전장치다.

**Independent Test**: 커버가 로드되는 동안 스피너가 아니라 커버 크기와 동일한 스켈레톤
placeholder가 보이고, 이미지 도착 시 아래 콘텐츠 위치가 밀리지 않는지 확인하는 것만으로
독립적으로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 게시글 상세 화면을 막 연 시점(커버 이미지 로드 미완료), **When** 화면이
   렌더링되면, **Then** 커버 자리에 스피너가 아닌 스켈레톤 placeholder가 표시된다.
2. **Given** 스켈레톤이 표시된 상태, **When** 커버 이미지 로드가 완료되면, **Then**
   스켈레톤이 실제 고양이 사진으로 대체되고 아래의 제목·본문 위치는 이동하지 않는다.
3. **Given** 스켈레톤이 차지한 영역, **When** 이미지가 로드되기 전과 후를 비교하면,
   **Then** 커버 영역의 크기(높이)가 동일하게 유지된다.

---

### User Story 3 - 커버 이미지를 불러오지 못했을 때 (Priority: P3)

외부 이미지 API가 응답하지 않거나 이미지를 가져오지 못하는 경우에도, 사용자는 깨진
이미지나 무한 스켈레톤에 갇히지 않고 편집 화면을 정상적으로 계속 사용할 수 있다.

**Why this priority**: 외부 오픈 API 의존은 실패 가능성을 내포한다. 실패가 편집이라는
핵심 작업을 막지 않도록 보장하는 견고성 요구사항이며, 핵심 표시(P1)의 하위 보호막이다.

**Independent Test**: 이미지 요청이 실패하는 상황을 만들었을 때, 깨진 이미지 아이콘이나
멈춘 스켈레톤이 남지 않고 제목·본문 편집이 정상 동작하는지 확인하는 것만으로 독립적으로
검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 커버 이미지 요청이 실패한 상태, **When** 화면이 이를 감지하면, **Then**
   깨진 이미지나 무한 로딩 스켈레톤 대신 커버 없이 정돈된 상태(빈 커버 영역 또는 커버
   미표시)로 정리되고 편집 화면은 계속 사용할 수 있다.
2. **Given** 커버 로드가 실패한 상태, **When** 사용자가 제목·본문을 입력하면, **Then**
   편집은 아무 영향 없이 정상 동작한다.

---

### Edge Cases

- **로딩 지연 / 무한 대기**: 이미지 응답이 매우 느리거나 끝내 오지 않을 때, 인위적
  타임아웃 없이 스켈레톤을 유지하며 화면의 나머지 부분(제목·본문 편집)은 즉시 사용
  가능해야 한다. 커버를 숨기는 것은 브라우저의 이미지 로드 에러가 발생한 경우로 한정한다.
- **요청 실패/타임아웃**: API가 오류를 반환하거나 응답이 없을 때 무한 스켈레톤이나 깨진
  이미지가 남지 않아야 한다 (User Story 3).
- **화면 재진입**: 사용자가 다른 화면으로 이동했다가 편집 화면으로 돌아올 때 커버 영역이
  일관되게 다시 표시되어야 한다 (매번 새 랜덤 이미지 — Assumptions 참조).
- **여러 게시글 이동**: 게시글 목록에서 서로 다른 게시글을 연속으로 열 때 각 편집 화면에
  커버 영역이 정상적으로 나타나야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 게시글 편집(상세) 화면에서 제목 입력창 바로 위에 커버 이미지
  영역을 배치해야 한다(MUST).
- **FR-002**: 시스템은 랜덤 고양이 사진 오픈 API로부터 커버 이미지를 가져와 표시해야
  한다(MUST). (엔드포인트: `https://cataas.com/cat/cute` — Assumptions 참조)
- **FR-003**: 커버 이미지를 불러오는 동안 시스템은 로딩 스피너가 아닌 스켈레톤
  placeholder를 커버 자리에 표시해야 한다(MUST).
- **FR-004**: 커버 이미지 로드가 완료되면 시스템은 스켈레톤을 실제 이미지로 대체해야
  하며, 이 전환으로 인해 아래 콘텐츠(제목·본문)의 위치가 이동해서는 안 된다(MUST) —
  즉 스켈레톤과 실제 커버는 동일한 영역(높이)을 차지해야 한다.
- **FR-005**: 커버 이미지 로드가 실패하면 시스템은 깨진 이미지나 멈춘 스켈레톤을 남기지
  않고 정돈된 상태(커버 없이)로 처리해야 하며, 편집 화면의 나머지 기능은 계속 동작해야
  한다(MUST).
- **FR-006**: 커버 이미지 영역은 제목·본문 편집을 방해하거나 가려서는 안 되며, 사용자는
  커버 유무·로딩 상태와 무관하게 제목과 본문을 정상적으로 편집할 수 있어야 한다(MUST).
- **FR-007**: 커버 이미지 영역은 문서 상단 콘텐츠로서 제목·본문과 함께 스크롤되어야
  한다(MUST).
- **FR-008**: 커버 이미지는 편집 화면에 진입할 때마다 새로 요청되어야 하며(MUST),
  게시글별로 특정 이미지를 저장·고정하지 않는다. 같은 게시글을 다시 열면 다른 이미지가
  표시될 수 있다.
- **FR-009**: 시스템은 커버 이미지 로딩에 인위적 타임아웃을 두지 않는다(MUST). 커버
  표시 실패는 브라우저의 이미지 로드 에러로만 판정하며(FR-005 참조), 로드가 지연되는
  동안에는 스켈레톤을 유지하되 제목·본문 편집은 즉시 가능해야 한다.

### Key Entities

- **커버 이미지(Cover Image)**: 편집 화면 상단에 표시되는 일시적(transient) 시각 요소.
  외부 랜덤 고양이 API에서 화면을 열 때 가져오는 이미지이며, 별도로 영구 저장되지 않는다
  (Assumptions 참조). 상태는 "로딩 중(스켈레톤)", "표시됨", "실패(미표시)" 세 가지다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 편집 화면을 여는 100%의 경우에서 제목 입력창 위에 커버 영역이 나타난다
  (로드 성공 시 이미지, 로딩 중 스켈레톤, 실패 시 정돈된 빈 상태 중 하나로 항상 표시).
- **SC-002**: 커버 이미지 로딩 중에는 100% 스켈레톤 placeholder가 표시되며, 회전
  스피너는 한 번도 표시되지 않는다.
- **SC-003**: 커버 이미지가 스켈레톤에서 실제 이미지로 바뀔 때 제목·본문의 화면상 세로
  위치 이동(layout shift)이 발생하지 않는다(0px 이동).
- **SC-004**: 커버 이미지 로드 성공·실패·지연 어느 경우에도 사용자는 제목과 본문을
  즉시(추가 대기 없이) 편집할 수 있다.
- **SC-005**: 커버 이미지 로드가 실패한 경우에도 깨진 이미지 아이콘이나 무한 스켈레톤이
  화면에 남지 않는다.

## Assumptions

- **API 엔드포인트**: 커버 이미지는 사용자가 지정한 `https://cataas.com/cat/cute`
  오픈 API에서 가져온다. 이 API는 요청마다 랜덤한 "cute" 태그 고양이 이미지를 반환한다.
  별도 인증 키가 필요 없는 공개 API로 가정한다.
- **이미지 갱신 정책**: 커버는 편집 화면을 열 때마다 새로 요청되어 매번 다른 랜덤 고양이가
  표시될 수 있다(Clarifications에서 확정 — FR-008). 게시글별로 특정 이미지를 영구
  저장·고정하지 않는다. 게시글별 커버 고정/영구 저장은 이번 범위에서 제외한다.
- **사용자 조작 범위**: 이번 버전에서 커버는 자동 표시만 하며, 사용자가 커버를 직접
  교체·제거·업로드하는 조작(예: "커버 변경" 버튼)은 범위에서 제외한다.
- **배치 대상 화면**: "제목 입력창"은 게시글 상세/편집 화면(`app/(app)/posts/[id]`)의
  제목 입력 필드를 가리키며, 커버는 그 위에 놓인다.
- **시각 규격**: 커버 영역의 크기·비율·라운드·여백 등 구체적 스타일 값은 프로젝트
  디자인 시스템(`DESIGN.md`)의 토큰·패턴을 따르며, 문서에 없는 결정이 필요하면
  `DESIGN.md`에 함께 반영한다(저장소 규칙 — 계획/구현 단계에서 확정).
- **네트워크**: 사용자는 외부 이미지 API에 접근 가능한 네트워크 환경을 가진다고 가정하며,
  접근 불가 상황은 로드 실패(User Story 3)로 처리한다.
